# Отчёт по логу запуска (5 апреля 2026)

Разобрал лог при запуске приложения. Нашёл 7 штук, из которых 3 реально важных, 2 — неприятных но терпимых, и 2 — просто шум от системы.

---

## 1. Шрифты не найдены — надо починить

**Что в логе:** FontParser ругается, что не может открыть 6 файлов шрифтов — ReenieBeanie, BigShouldersStencil, CarterOne, Tourney, UnifrakturCook, VastShadow.

**Что на самом деле:** Файлов шрифтов (`.ttf`) в проекте нет вообще — ни одного. Папка `StepsTrader/Fonts` в Xcode пустая — группа есть, файлов нет. При этом в `Info.plist` все 6 шрифтов прописаны в `UIAppFonts`, поэтому система при запуске их ищет и не находит.

В коде (`Font+Custom.swift`) шрифты используются через хелперы: `.reenie()`, `.bigShouldersStencil()` и т.д. У большинства есть фолбэк на системный шрифт (если `UIFont(name:)` вернёт nil), а вот у `.reenie()` фолбэка нет — он всегда пойдёт через `.custom("ReenieBeanie", ...)`, и если шрифт не найден, SwiftUI подставит дефолтный, но без контроля.

**Что делать:**
- Либо добавить `.ttf` файлы в проект + в Copy Bundle Resources
- Либо, если шрифты больше не нужны, убрать их из `Info.plist` и почистить код

## 2. Дублирующийся ID в ForEach — надо починить

**Что в логе:** SwiftUI ругается: `the ID about_colors occurs multiple times within the collection`.

**Что на самом деле:** В файле `Note.swift` в каталоге заметок (`NoteCatalog.all`) есть две заметки с одинаковым `id: "about_colors"`. Тексты у них разные (одна про то, как выбирали цвета, другая — про то, что цвета не продаются), но идентификатор один и тот же.

Проблема всплывает в `ManualsPage` — и в `TabView` (который показывает карточки), и в списке всех заметок (`AllNotesListView`). Оба используют `ForEach` по `Note.id`, а там дубликат.

Побочные эффекты от дубликата:
- Если прочитать одну заметку «About Colors», вторая тоже пометится как прочитанная (трекер привязан к id)
- При клике на вторую заметку в списке, откроется первая (поиск по `firstIndex(where: { $0.id == note.id })` всегда найдёт первую)
- Случайная заметка (`NoteCatalog.random`) исключит обе разом

**Что делать:** Дать второй заметке уникальный id, например `about_colors_economy` или `about_colors_2`.

## 3. Очередь из 50 запросов, которые не уходят — надо починить

**Что в логе:**
```
Draining 50 queued sync requests
Retry queue drained: 0 succeeded, 50 still pending
Analytics flush failed: HTTP 409
```

**Что на самом деле:** В `SupabaseSyncService` есть очередь для запросов, которые не прошли (офлайн или ошибка). Максимум 50 штук. При запуске приложения очередь пытается «слить» все накопившиеся запросы. Но ни один из 50 не проходит.

Почему не проходят: когда запрос попадает в очередь, сохраняются только URL, метод и тело. **Заголовок `Prefer: resolution=merge-duplicates` не сохраняется.** А именно он нужен Supabase, чтобы сделать upsert (вставить-или-обновить). Без него повторная отправка — это просто INSERT, который падает с 409 (конфликт), потому что запись с таким ключом уже есть.

А раз 409 не входит в список `retryableStatusCodes` (там только 408, 429, 500, 502, 503, 504), то NetworkClient даже не пытается повторить — сразу возвращает ошибку, запрос остаётся в очереди, и так по кругу.

Отдельно: аналитика падает с 409 по похожей причине — возможно, дубли событий или конфликт на уровне таблицы.

**Что делать:** При сохранении запроса в очередь нужно сохранять и заголовок `Prefer` (или хотя бы тип запроса, чтобы при воспроизведении добавить нужный заголовок). Иначе всё, что упало в оффлайне, никогда не дойдёт.

## 4. Картинки сохраняются с ненужным альфа-каналом — мелочь, но жрёт память

**Что в логе:** `writeImageAtIndex: ERROR: 'Nowhere' is trying to save an opaque image (400x400) with 'AlphaPremulLast'` — повторяется 4 раза.

**Что на самом деле:** Это `CanvasStorageService.saveWidgetSnapshot()`. Он рендерит `GenerativeCanvasView` в картинку 200x200 точек с масштабом 2 — получается 400x400 пикселей. Сохраняет через `jpegData()`. Проблема в том, что `ImageRenderer` отдаёт буфер с альфа-каналом (RGBA), а JPEG альфу не поддерживает. Система ругается и альфу отбрасывает.

Само по себе ничего не ломает — картинка сохранится. Но:
- Расход памяти при декодировании удваивается (система сначала декодирует с альфой, потом убирает)
- Файл чуть больше, чем мог бы быть
- 4 раза за один запуск — потому что при запуске состояние канваса обновляется несколько раз и каждый раз триггерит `saveWidgetSnapshot`

**Что делать:** Можно из `UIImage` достать `cgImage` и пересоздать без альфы перед сохранением. Или уменьшить количество вызовов `saveWidgetSnapshot` при запуске (дебаунс или флаг «уже сохранили»). Не горит, но хорошая оптимизация.

## 5. HealthKit «sharing denied» — всё нормально, не баг

**Что в логе:** `steps=sharing denied, sleep=sharing denied` — повторяется много раз.

**Что на самом деле:** Это **нормально**. Приложение запрашивает только чтение (`toShare: []`, `read: [steps, sleep, workouts, mindful]`). Функция `authorizationStatus(for:)` в HealthKit показывает статус **записи**, а не чтения. Для приложения, которое только читает, `sharingDenied` — это ожидаемый ответ.

Данные при этом приходят — 15709 шагов, 7.69 часов сна — всё работает. В коде даже есть комментарии, объясняющие это (в `HealthStore.swift` и `HealthKitService.swift`).

Единственное — лог выглядит пугающе. Можно переименовать в логе на что-то типа «write=denied (expected for read-only)» чтобы не путаться.

## 6. UserDefaults предупреждение про контейнер — шум от системы

**Что в логе:** `Couldn't read values in CFPrefsPlistSource ... Using kCFPreferencesAnyUser with a container is only allowed for System Containers, detaching from cfprefsd`.

**Что на самом деле:** App Group настроен правильно — `group.personal-project.StepsTrader` прописан в entitlements всех таргетов (приложение, виджет, DeviceActivityMonitor, ShieldConfiguration, ShieldAction). В коде группа используется через `SharedKeys.appGroupId` и `UserDefaults(suiteName:)`.

Это сообщение — внутренняя кухня `cfprefsd`. Система пробует одну стратегию доступа, она не подходит для app group (не системный контейнер), и переключается на другую. Данные при этом нормально читаются и пишутся.

Стоит обратить внимание, только если:
- `UserDefaults(suiteName:)` вдруг вернёт nil
- Расширения перестанут видеть данные из основного приложения
- Значения будут теряться между запусками

Пока таких симптомов нет — можно игнорировать.

## 7. [carc] Reporter disconnected — полный шум

**Что в логе:** `[carc] Reporter disconnected or already stopped` и `Message send exceeds rate-limit threshold and will be dropped` — повторяются много раз.

**Что на самом деле:** Это Core Analytics (системная штука Apple). Никакого отношения к коду приложения не имеет. Предупреждение о rate-limit говорит о том, что что-то генерирует много аналитических событий — скорее всего, те самые 50 запросов в очереди и сетевая активность при запуске. Само по себе ничего не ломает.
